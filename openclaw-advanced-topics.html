主題07

多重代理指揮官 —— 同時操控多個 Gateway

練習在同一台主機上啟動兩個不同設定、且完全隔離的 AI 背景服務。

延伸挑戰 A
🧰 事前準備：確保你能在終端機正常呼叫 openclaw。這個主題會用到環境變數來指定不同的設定檔與狀態目錄，讓兩個閘道互不干涉。

● 基礎實作 · 設定與執行多個閘道實例

  1. 1

    先幫第二個閘道準備一份獨立設定檔。直接複製一份你現有的設定，改名為 b.json：

    cp ~/.openclaw/openclaw.json ~/.openclaw/b.json

    👀 你會看到：指令沒有任何輸出就是成功了。現在 ~/.openclaw/ 底下多了一個 b.json。

    ✅ 確認：輸入 ls ~/.openclaw/，清單裡看得到 b.json。

    💡 如果你的主設定檔叫別的名字（例如 config.json），把上面的來源檔名換掉即可；不確定的話對 Agent 說「幫我找出 openclaw 的設定檔在哪」。

  2. 2
    openclaw gateway --port 19001

    👀 你會看到：閘道在前景啟動，並在連接埠 19001 開始監聽。它會輸出啟動日誌，表示正在等待客戶端連線。

    ✅ 確認：終端機停在此畫面，沒有報錯，且顯示類似 Listening on 127.0.0.1:19001 的訊息。

  3. 3

    開啟第二個終端機視窗，在新的視窗中輸入以下指令，用剛剛建立的 b.json 與一個獨立狀態目錄，啟動第二個隔離的閘道：

    OPENCLAW_CONFIG_PATH=~/.openclaw/b.json OPENCLAW_STATE_DIR=~/.openclaw-b openclaw gateway --port 19002

    👀 你會看到：第二個閘道成功在連接埠 19002 啟動。因為使用了獨立的狀態與設定路徑，它與第一個閘道是完全互不干涉的。

    ✅ 確認：兩個終端機視窗都在執行閘道，且分別佔用 19001 與 19002 埠。

  4. 4

    開啟第三個終端機視窗，使用系統診斷指令來查看目前主機上所有運作中的閘道服務：

    openclaw gateway status --deep

    👀 你會看到：終端機列出當前系統中偵測到的所有 OpenClaw 相關服務，包括我們剛剛手動開啟的實例，並會顯示是否有設定偏移（drift）的警告。

    ✅ 確認：輸出中有列出我們的背景服務狀態，並且能清楚識別多個執行個體的連接埠。

💡 熱重新載入（Hot Reload）：
OpenClaw 的閘道預設啟用 gateway.reload.mode="hybrid"。這意味著當你變更設定檔（例如 b.json）時，執行中的程序會自動監看並以無縫（atomic）方式在記憶體中置換快照，不需要你每次手動重開程序。
卡住了？
① 出現 another gateway instance is already listening 或 EADDRINUSE 錯誤 → 代表你要使用的連接埠已經被佔用了。請用 openclaw gateway stop 停止舊的服務，或換一個連接埠（如 19003）再試一次。
② 執行系統診斷時發現設定偏移或有遺漏 → 輸入 openclaw doctor 進行自動修復。

● 進階挑戰 · 讓兩個閘道掛不同的大腦，同場較勁

一個用雲端 Ollama Cloud，一個用本機 Ollama

既然兩個閘道完全隔離，就讓它們各自接不同的模型：19001 沿用你 Day 1 裝好的 Ollama Cloud 雲端模型，19002 改接本機離線的 Ollama。同一個問題丟給兩邊，比較「雲端 vs 本機」的差異。

  1. 1

    編輯 b.json，把它的預設模型改成本機 Ollama（例如 ollama/llama3.2:1b）。不確定怎麼改就對第二個閘道的 Agent 說：

    把你的預設模型換成本機 Ollama 的 llama3.2:1b，改好後告訴我要不要重啟。

    ✅ 確認：19001 仍走 Ollama Cloud 雲端模型、19002 走本機 Ollama，兩邊各自回報自己現在用的模型。

  2. 2

    對兩個閘道問同一個需要即時知識的問題（例如「2024 巴黎奧運在哪一天開幕？」），比較誰答得準、誰比較快。

    ✅ 確認：你能說出兩者至少一個差異（正確性、速度、或是否需要連網），並解釋為什麼。

主題08

自製 AI 服務端點 —— 串接 OpenAI 相容 API

將 OpenClaw Gateway 為 API 伺服器，利用標準 OpenAI 協定來連接外部軟體或程式。

延伸挑戰 B
🧰 事前準備：① 確保 OpenClaw 正在背景或前景執行（預設連接埠 18789）。② 找到你的 OpenClaw 金鑰（Token）。你可以用 TUI 介面或指令 openclaw dashboard --no-open 取得目前的 Token。③ 先開好跨來源許可，等一下才不會被瀏覽器擋下來——把 http://localhost 與 null 兩個來源加進設定檔的 gateway.controlUi.allowedOrigins。懶得手動改就對 Agent 說：
把 http://localhost 和 null 這兩個來源加進 gateway.controlUi.allowedOrigins，改好後告訴我要不要重啟。

● 基礎實作 A · 連接外部 Chat UI (以 LobeChat / Open WebUI 為例)

  1. 1

    開啟 LobeChat 或任何支援 OpenAI API 格式的客戶端，進入「設定 -> 語言模型 -> OpenAI」。

    將 API 代理地址 (Base URL) 設為：
    http://127.0.0.1:18789/v1

    將 API Key 填入你的 OpenClaw Token。

    ✅ 確認：點選「獲取模型清單」或測試連線時，客戶端成功抓取到 openclaw/default 模型。

  2. 2

    在客戶端中發送一句問候（如「哈囉，請告訴我你是誰」）。

    👀 你會看到：外部軟體成功接收到來自你本機 OpenClaw Agent 的回覆。OpenClaw 會將 /v1/chat/completions 的請求轉發給預設 Agent 來處理。

● 基礎實作 B · 寫幾行 Javascript 呼叫你自己的 Agent

  1. 3

    在桌面建立一個名為 chat.html 的檔案，使用記事本或編輯器打開，貼入以下程式碼（記得將 Bearer 後方的 TOKEN 替換為你的 OpenClaw Token）：

    <!DOCTYPE html>
<html>
<body>
  <h2>呼叫本機 Agent 測試</h2>
  <button onclick="callAgent()">發送測試請求</button>
  <p id="output">等待回應...</p>

  <script>
    async function callAgent() {
      const el = document.getElementById('output');
      el.innerText = '思考中...';
      try {
        const res = await fetch('http://127.0.0.1:18789/v1/chat/completions', {
          method: 'POST',
          headers: {
            'Content-Type': 'application/json',
            'Authorization': 'Bearer YOUR_OPENCLAW_TOKEN_HERE'
          },
          body: JSON.stringify({
            model: 'openclaw/default',
            messages: [{ role: 'user', content: '用一句話問候我' }]
          })
        });
        const data = await res.json();
        el.innerText = data.choices[0].message.content;
      } catch (err) {
        el.innerText = '發生錯誤: ' + err.message;
      }
    }
  </script>
</body>
</html>

    雙擊打開 chat.html 網頁，並點擊「發送測試請求」按鈕。

    ✅ 確認：網頁畫面上成功顯示了 Agent回傳的問候語。這代表你已經能用語法控制你專屬的 Agent 運作了！

  2. 4

    此時回到你啟動 OpenClaw Gateway 的終端機視窗，或者另開一個終端機執行：

    openclaw logs --follow

    👀 你會看到：終端機中實時滾動輸出剛才網頁與 LobeChat 傳送過來的 HTTP 請求紀錄。這就是日誌（Logs）除錯的魅力，你可以看到所有對話與工具呼叫的底層運作軌跡！

卡住了？
① 連線時出現 unauthorized 或 401 錯誤 → 代表你的 Token 填錯了，請重新複製正確的 Token。
② 瀏覽器中提示 CORS 或來源不允許錯誤 → 你八成漏了「事前準備 ③」。請確認網頁的來源（雙擊開啟的檔案來源是 null，本機伺服器則是 http://localhost）已加進 gateway.controlUi.allowedOrigins，並重啟閘道。

● 進階挑戰 · 用程式語言呼叫，並讓回覆一個字一個字跳出來

Python 三行呼叫 ＋ 串流（streaming）回應

網頁只是開胃菜。真正的軟體幾乎都用後端語言呼叫 API。把下面存成 chat.py，記得換掉 Token，然後在終端機執行 python3 chat.py：

import requests

res = requests.post(
    "http://127.0.0.1:18789/v1/chat/completions",
    headers={"Authorization": "Bearer YOUR_OPENCLAW_TOKEN_HERE"},
    json={
        "model": "openclaw/default",
        "messages": [{"role": "user", "content": "用一句話介紹你自己"}],
    },
)
print(res.json()["choices"][0]["message"]["content"])

✅ 確認：終端機印出 Agent 的一句話回覆。恭喜，你剛剛用程式語言驅動了自己的 AI。

再加難：在送出的 JSON 裡多加一行 "stream": True，改用 requests.post(..., stream=True) 並逐行讀 res.iter_lines()，讓回覆像打字機一樣一段一段跳出來。卡住就把這段需求整個貼給 Agent，請它幫你改寫並解釋每一行。

主題09

AI 雲端任意門 —— 跨網路遠端存取與安全性

理解 OpenClaw 的安全繫結邊界，並使用 SSH 隧道或 Tailscale 進行安全的跨網連線。

延伸挑戰 C
🧰 事前準備：① 了解你的主機 IP 位址。② 需要有一台可以透過 SSH 連線的遠端主機，或是安裝了 Tailscale。只有一台電腦？沒關係——這個主題最適合兩人一組：一人當「主機方」開閘道，另一人當「遠端方」連過去，角色互換再玩一次。
🔒 為什麼 OpenClaw 預設只聽 localhost？
OpenClaw 的 Agent 具有讀寫你本機電腦檔案與執行終端機指令的權限。如果你將 Gateway 隨意繫結（Bind）到 0.0.0.0 且沒有設定強效密碼，網路上任何人都可以隨意控制你的電腦。因此，預設的繫結模式是 loopback（即 127.0.0.1），只允許本機連線。要跨網連線，不是把門大開，而是挖一條只有你進得來的加密地道——這就是隧道技術。

● 實作練習 A · 建立 SSH 安全加密隧道（不必動繫結設定）

  1. 1

    若你要從遠端電腦連回主機方的 OpenClaw 閘道，請在遠端電腦執行以下 SSH 隧道指令（將 user@gateway-host 換成主機方的 SSH 登入資訊）：

    ssh -N -L 18789:127.0.0.1:18789 user@gateway-host

    👀 原理說明：這個指令會在你的遠端電腦上監聽 18789 埠，並把所有流向此埠的流量，透過加密的 SSH 通道，安全地轉發到主機方的 127.0.0.1:18789。對閘道來說，連線「看起來」還是來自本機，所以完全不用改繫結設定，也不必暴露公網 IP！

    ✅ 確認：遠端電腦的終端機停住不動、沒有報錯，代表隧道已經建立成功。

  2. 2

    隧道開著別關，在遠端電腦的瀏覽器輸入控制台網址 http://127.0.0.1:18789，並貼上主機方給你的 Token。

    👀 你會看到：控制台順利打開。你操作的雖然是本機的 18789，實際上經由地道連到的是另一台電腦的 Agent。

    ✅ 確認：在遠端電腦上成功操控了另一台電腦的 AI Agent，且主機方的閘道終端機看到 SSH 轉接的連線日誌。

● 實作練習 B · 體驗零設定的 VPN 加密網絡 —— Tailscale

  1. 3

    如果不想手動打 SSH 指令，可以在兩台電腦上都下載並安裝 Tailscale 並登入同一個帳號。啟用後，每台電腦都會拿到一個 100.xx.xx.xx 的虛擬私人 IP。

    ⚠️ 和 SSH 隧道不同，這裡需要主機方動一次繫結設定。因為預設的 loopback 只聽 127.0.0.1，Tailscale 的流量進不來。請主機方把繫結改成也監聽 Tailscale 介面——最省事的方式是對 Agent 說：

    把 OpenClaw 閘道的繫結改成也允許我的 Tailscale 網路連線（tailnet），其他來源維持擋掉。改好後告訴我要不要重啟，以及我該用哪個網址連。

    👀 安全提醒：Tailscale 是私有網路，只有登入同一帳號的裝置進得來；但務必不要改成 0.0.0.0，那等於對整個公網開門。

    ✅ 確認：在另一台裝置的瀏覽器輸入 http://<主機方的 Tailscale-IP>:18789，輸入 Token 後成功打開控制台。

● 進階挑戰 · 把教室的電腦裝進口袋

用手機透過 Tailscale 遠端指揮教室主機的 Agent

把上面的「遠端方」換成你的手機：在手機安裝 Tailscale App 登入同一帳號，就能在回家的路上，用手機瀏覽器連回教室那台電腦的 Agent，叫它幫你整理今天的檔案。

  1. 1

    手機裝好 Tailscale、登入同一帳號並確認連線亮綠燈；主機方維持練習 B 改好的 tailnet 繫結。

    ✅ 確認：手機瀏覽器開 http://<主機方 Tailscale-IP>:18789 能看到控制台登入畫面。

  2. 2

    用手機登入後，對教室主機的 Agent 下一個真的會動到那台電腦的指令：

    把你 workspace 裡今天新增的檔案列出來，整理成一則三行的摘要給我。

    ✅ 確認：手機上收到的回覆，反映的是教室那台電腦的實際檔案——你已經把一台電腦裝進口袋了。

安全守則防線：
即使使用了 SSH 隧道或 Tailscale，OpenClaw 的驗證機制依然在起作用。請勿為了貪圖方便而將 gateway.auth.token 設為空值，也不要把繫結改成 0.0.0.0，否則一旦通道被開啟，仍會存在安全風險。
